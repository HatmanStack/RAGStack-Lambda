# Phase 0: Foundation

## Architecture Decisions

### ADR-1: Async Lambda Invocation (not Step Functions, not SQS)

The AppSync resolver validates input and calls `lambda_client.invoke(InvocationType='Event')` on QueryKBFunction. This matches the existing pattern used at three points in `src/lambda/appsync_resolvers/index.py` (image processing, KB ingestion, batch processing). Step Functions and SQS are overkill for a single-step operation.

### ADR-2: Polling over WebSocket Subscriptions

The ragstack-chat web component has zero runtime dependencies. Implementing AppSync realtime WebSocket protocol (IAM SigV4 WSS handshake, keep-alive, reconnection) would add disproportionate complexity. Polling 10-15 lightweight DynamoDB reads over 30-50s is trivial. Fixed 2s polling interval (not exponential backoff) since the expected wait is 30-50s and the reads are inexpensive.

### ADR-3: Store Results in Existing ConversationHistoryTable

No new DynamoDB table. The existing table has `conversationId` (HASH) + `turnNumber` (RANGE). The query_kb handler already writes conversation turns here. The polling query reads from the same table. A new `requestId` attribute is added to each turn record so the client can correlate which response matches which sent message.

### ADR-4: ConversationHistoryTable Schema Extension

The existing schema (`conversationId` PK, `turnNumber` SK) works without modification. Each turn already stores `userMessage`, `assistantResponse`, `sources`, `createdAt`, `ttl`. We add:
- `requestId` (String) - per-message correlation ID, generated by client
- `status` (String) - `PENDING` or `COMPLETED` or `ERROR`
- `errorMessage` (String, optional) - error details when status is `ERROR`

The mutation resolver writes a PENDING record (with `userMessage` only). The query_kb handler updates it to COMPLETED (with `assistantResponse`, `sources`) or ERROR. The `getConversation` query reads all turns for a conversationId.

### ADR-5: Replace Sync Query Entirely

No backward compatibility path. The sync `queryKnowledgeBase` query is removed and replaced with the mutation. Nothing outside the dashboard and ragstack-chat calls this API.

### ADR-6: Client-Generated IDs

The client generates both `conversationId` (UUID, on first message, reused for subsequent turns) and `requestId` (UUID, per message). This avoids a round-trip to get server-generated IDs and matches the existing pattern where `conversationId` is already client-generated.

### ADR-7: Concurrent Message Handling

Block concurrent sends in the UI. If a response is pending, the input is disabled. This is simpler than handling multiple in-flight requests and matches user expectations (chat is sequential).

## Shared Patterns

### Async Lambda Invocation Pattern (existing)
```python
lambda_client.invoke(
    FunctionName=FUNCTION_ARN,
    InvocationType="Event",  # Async invocation
    Payload=json.dumps(event).encode(),
)
```
Used at `src/lambda/appsync_resolvers/index.py` lines 837, 1341, 2350.

### AppSync Resolver Routing Pattern (existing)
The `lambda_handler` in `appsync_resolvers/index.py` routes by `event["info"]["fieldName"]` to handler functions via a `resolvers` dict. New resolvers follow this pattern.

### DynamoDB Conversation Record Shape
```python
{
    "conversationId": "uuid-string",     # PK
    "turnNumber": 1,                      # SK (integer)
    "requestId": "uuid-string",           # NEW: per-message correlation
    "status": "PENDING|COMPLETED|ERROR",  # NEW: async status
    "userMessage": "...",
    "assistantResponse": "...",           # Empty for PENDING
    "sources": "[]",                      # JSON string, empty for PENDING
    "errorMessage": "...",                # Only for ERROR status
    "createdAt": "ISO-8601",
    "ttl": 1234567890,
}
```

### GraphQL Auth Directives
New mutation and query need `@aws_iam @aws_api_key @aws_cognito_user_pools` to match the existing `queryKnowledgeBase` auth pattern (ragstack-chat uses IAM via Cognito Identity Pool, external integrations use API key, admin UI uses Cognito).

## Testing Strategy

### Python Unit Tests
- Location: `tests/unit/python/`
- Framework: pytest
- Run: `uv run pytest tests/unit/python/ -v`
- Mocking: `unittest.mock.patch` for boto3 clients, DynamoDB, Lambda
- Existing test files: `test_query_kb.py`, `test_query_kb_media.py`

### Frontend Unit Tests
- Location: `src/ragstack-chat/src/components/__tests__/`
- Framework: Vitest
- Run: `npm run test:frontend`
- Mocking: `vi.fn()` for `iamFetch`, `fetchCDNConfig`
- Existing test file: `ChatInterface.test.tsx`

### No Integration Tests Required
Integration tests require a deployed stack. This plan covers unit tests only. The existing `@pytest.mark.integration` tests in `tests/integration/` will be updated separately after deployment.
